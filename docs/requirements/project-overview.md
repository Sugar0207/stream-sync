# StreamSync Project Overview

## 1. プロジェクト概要
StreamSync は、4人のゲーム映像を受信し、共通の時間軸で同期したうえで、OBS に載せられる形で表示・手動切り替えできる多視点スイッチング配信ツールです。

主な用途は、Minecraft などのゲームを複数人で配信する際に、各参加者の映像をほぼ同じ時間基準でそろえ、4分割表示や単独表示を切り替えながら YouTube ライブ等へ配信することです。

---

## 2. 目的
このプロジェクトの目的は、4人のゲーム映像を完全同期に近い形でそろえ、実際の配信で使える実用的なスイッチング基盤を作ることです。

特に重視するのは以下です。

- 4人分の映像を共通の時間軸でそろえること
- OBS に載せられる安定した表示を作ること
- 手動で視点を切り替えられること
- PoC から MVP へ段階的に育てられること
- 将来的に 1080p / 60fps へ拡張可能な構造を持つこと

---

## 3. 想定利用シーン
- 4人グループでの Minecraft 配信
- 複数視点を見せたい企画配信
- 逃走中、PVP、班行動、役割分担がある企画
- 同時多発的に見どころが起きる配信
- 4分割表示と単独表示を切り替える配信

---

## 4. MVPで重視すること
MVP では次の点を優先します。

- 完全同期に向けた映像基盤
- 4人固定での安定運用
- 720p / 30fps を標準とした安定動作
- OBS との実用的な連携
- 手動スイッチング
- 遅延や同期状態の可視化
- 実運用に耐えるログと計測

---

## 5. MVPでやらないこと
MVP では以下を対象外とします。

- 音声を専用ツールに統合すること
- 自動スイッチング
- 発話検知による自動強調
- Minecraft イベント連動演出
- 録画保存やアーカイブ管理
- リプレイ機能
- クリップ自動生成
- 5人以上への一般化
- 視点数の動的増減対応
- 高度な権限管理
- 一般公開向けの完成品品質までの仕上げ
- OBS の高度な自動制御

---

## 6. PoC 完了条件
PoC の完了条件は以下です。

- 1人分のゲーム画面をキャプチャできる
- 1人分の映像を別プロセスまたは別PCへ送信できる
- 受信側でその映像を表示できる
- フレームにタイムスタンプを付与できる
- 2人分の映像を同時に受信して並べて表示できる
- 2人分の映像を同じ時間基準でおおまかにそろえられる
- 4人分の映像を同時に受信して4分割表示できる
- 各視点の遅延や接続状態を確認できる
- targetTime ベースで各視点の表示フレームを選べる
- 4視点を安定表示できることを短時間テストで確認できる
- OBS で取り込める形で表示できる

---

## 7. MVP 完了条件
MVP の完了条件は以下です。

- 4人のクライアントが安定して接続できる
- 各クライアントの映像を継続的に送信できる
- 受信側で4視点を同時に受信・復号・表示できる
- 各視点を共通の時間軸で同期表示できる
- 固定遅延量を設定してジッターバッファを運用できる
- 遅延・jitter・接続状態を可視化できる
- 4分割表示と単独表示を切り替えられる
- 手動スイッチングができる
- ホットキーで視点切り替えできる
- 切断中・未接続・復号失敗などの状態を画面上で判別できる
- OBS で安定して取り込める
- 配信用表示として不要 UI を隠せる
- 30分〜1時間程度の連続運用に耐えられる
- クライアント切断後の再接続に対応できる
- 最低限の認証付きで許可クライアントのみ接続できる
- ログから遅延や切断の原因を追える
- Minecraft 実機で 4人テストを行い、実用可能と判断できる
- 起動手順・接続手順・配信前チェック手順が用意されている

## 8. 決定済み技術方針
### 8.1 技術スタック
- 言語: Rust
- 映像処理: FFmpeg 系
- 通信: UDP 独自プロトコル
- コーデック: H.264
- UI: Rust 製の最小 GUI
- OBS 連携: switcher 専用ウィンドウを Window Capture
- 設定ファイル: TOML
- ログ: JSON Lines 形式の構造化ログ

### 8.2 認証
- 事前共有トークン方式
- clientId ホワイトリスト
- 認証済みクライアントの UDP パケットのみ受理

### 8.3 バージョン管理
- app_version と protocol_version を分離する
- protocol_version 不一致の client は接続拒否する
- app_version 差異は warn ログとして扱う

### 8.4 ネットワーク構成
- client 4台が中央 server に直接 UDP 送信するスター構成
- server が同期責任を持つ
- switcher は表示専用
- MVP 初期段階では server と switcher は同一 PC 運用でよい

### 8.5 音声方針
- MVP では Discord を継続使用する
- 配信用音声は 1 系統に固定する
- 視聴者向け映像は基準音声に合わせて遅延調整する
- スイッチャー監視用音声は低遅延のまま扱う
- 音声統合は将来拡張とする

## 9. 標準品質と将来拡張
### 9.1 標準品質
- 720p / 30fps を標準運用とする

### 9.2 将来拡張
- 1080p / 60fps に上げられる設計にする
- 1080p / 60fps は条件付き上位運用モードとする
- 実装は設定駆動で行い、固定値ベタ書きを避ける
- ハードウェアエンコードを強く前提にする
