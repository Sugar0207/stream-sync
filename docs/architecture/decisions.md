# StreamSync Architecture Decisions

## 目的

このドキュメントは、StreamSync における主要な設計判断と、その採用理由を記録するためのものです。

目的は以下です。

- なぜその技術や構成を選んだかを明確に残す
- 後から同じ議論を繰り返さないようにする
- Codex や将来の開発時に、既存方針を崩しにくくする
- 仕様変更が必要になった場合に、変更影響を判断しやすくする

---

## ADR-001: プロジェクト名を StreamSync とする

### 決定
プロジェクト名は `StreamSync` とする。  
リポジトリ名 / ルートフォルダ名は `stream-sync` とする。

### 理由
- 配信 (`Stream`) と同期 (`Sync`) の中心価値が分かりやすい
- 今回の用途に直結している
- 将来 4人固定以外へ拡張しても破綻しにくい
- ディレクトリ名や README に置いたときに自然

### 影響
- リポジトリ名、README、ドキュメント名の基準になる
- アプリ表示名やログ表示でも使いやすい

---

## ADR-002: 完全同期を最優先目標とする

### 決定
このプロジェクトでは、一般的な多視点配信よりも **完全同期に近い映像表示** を最優先目標とする。

### 理由
- このツールの独自価値は、4人の映像をほぼ同じ時間基準でそろえることにある
- 単なる画面切り替えは OBS でもある程度可能
- 差別化ポイントは「同期基盤」にある

### 影響
- 通信方式、同期方式、バッファ設計はこの方針を最優先にする
- UI や演出より、まず同期の成立を優先する
- 高画質化より同期安定性を優先する

---

## ADR-003: MVP では 4人固定とする

### 決定
MVP では対象人数を 4人固定とする。

### 理由
- 想定利用シーンが 4人グループの配信である
- 可変人数対応を最初から入れると、設計が不要に複雑になる
- 4分割表示、固定役割、固定 UI のほうが PoC / MVP に向いている

### 影響
- UI は 2x2 レイアウトを前提にしてよい
- 設定も 4人前提で単純化できる
- 5人以上対応は将来拡張とする

---

## ADR-004: 実装言語は Rust を採用する

### 決定
主要コンポーネントの実装言語は Rust とする。

### 理由
- リアルタイム処理に向いている
- 並行処理や長時間運用に強い
- 同期、バッファ、受信、表示などの中核処理で性能面の安心感がある
- Node.js / Python より中核性能面で有利
- C++ より保守しやすいバランスを期待できる

### 影響
- apps / crates は Cargo workspace 前提で構成する
- FFmpeg 系ライブラリ連携やネイティブ寄り実装を前提にする
- Web UI 中心構成は採用しない

---

## ADR-005: 通信方式は UDP 独自プロトコルを採用する

### 決定
client と server 間の通信は UDP 独自プロトコルを採用する。

### 理由
- 低遅延を優先したい
- 古いフレームは再送より破棄したい
- server 側で targetTime ベースの同期を行いたい
- TCP のような詰まりを避けたい
- WebRTC / SRT / RIST より、同期ロジックの主導権を持ちやすい

### 不採用にしたもの
- TCP
- WebRTC
- SRT
- RIST
- RTP/RTCP over UDP

### 影響
- 認証、heartbeat、メッセージ定義を自前で持つ
- protocol_version 管理が重要になる
- 古いフレームの扱いは server 側設計に依存する

---

## ADR-006: コーデックは H.264 を採用する

### 決定
映像コーデックは H.264 を採用する。

### 理由
- 対応が広い
- FFmpeg 系との相性が良い
- ハードウェアエンコードが使いやすい
- 720p / 30fps でも 1080p / 60fps でも現実的
- H.265 / AV1 より MVP の安定運用に向いている

### 影響
- client 側のハードウェアエンコード対応を重視する
- payload の codec 識別は MVP では `h264` を前提にできる
- 将来別 codec へ広げる余地は残す

---

## ADR-007: 標準品質は 720p / 30fps とする

### 決定
MVP の標準品質は 720p / 30fps とする。

### 理由
- 4人同時運用として現実的
- クライアントの上り回線やエンコード負荷が過剰になりにくい
- PoC / MVP の標準として扱いやすい
- 同期基盤の成立確認を優先できる

### 影響
- 初期設定値は 720p / 30fps を中心に置く
- README や手順書でも標準運用として扱う
- 画質より安定性を優先する

---

## ADR-008: 1080p / 60fps は条件付き上位運用モードとする

### 決定
1080p / 60fps は将来拡張として設計するが、MVP の標準運用にはしない。条件付き上位運用モードとする。

### 理由
- 720p / 30fps と比べて約 4.5 倍重い
- クライアントの上り回線、GPU エンコード、受信側デコード負荷が大きく増える
- MVP の主目的は同期基盤の確立であり、高品質化ではない
- 実装は設定駆動にして、後で上げられるようにすれば十分

### 影響
- 解像度、fps、ビットレート、GOP などは設定化する
- ハードウェアエンコードを強く前提にする
- 上位運用条件を別途定義する

---

## ADR-009: OBS 連携は Window Capture を採用する

### 決定
OBS 連携方法は、switcher 専用表示ウィンドウを OBS の Window Capture で取り込む方式を採用する。

### 理由
- 構成が単純
- MVP に向いている
- switcher を表示専用アプリとして分離しやすい
- Browser Source や OBS プラグインより初期コストが低い
- 同期済み映像の専用ウィンドウを素直に配信ソースにできる

### 不採用にしたもの
- Browser Source を前提にした Web UI
- OBS プラグイン中心構成
- Virtual Camera 経由の連携

### 影響
- switcher は専用表示ウィンドウを持つ必要がある
- 将来、配信用表示と操作用表示を分ける可能性がある
- OBS 側設定は比較的単純になる

---

## ADR-010: 音声は MVP では Discord を継続使用する

### 決定
音声は MVP では専用ツールに統合せず、Discord を継続使用する。

### 理由
- 音声統合は実装コストが高い
- MVP で主役なのは映像同期基盤
- 通話基盤を今崩す必要がない
- 配信用音声の基準系統を決めて映像側で合わせる方が早い

### 影響
- 音声は protocol 対象外とする
- 配信用音声系統の定義が必要になる
- 映像側で遅延調整を行う

---

## ADR-011: ネットワーク構成は中央 server 方式を採用する

### 決定
4人の client が中央 server に直接 UDP 送信するスター構成を採用する。

### 理由
- 同期責任を一か所に集約できる
- targetTime ベースの制御を server に集中できる
- P2P より責務が明確
- 将来 switcher / server の分離や遠隔構成へ拡張しやすい

### 影響
- server が同期基盤の中心になる
- switcher は表示専用にできる
- client 間直接通信は MVP で不要になる

---

## ADR-012: 認証は事前共有トークン方式 + clientId ホワイトリストとする

### 決定
MVP では、事前共有トークン方式 + clientId ホワイトリストを採用する。

### 理由
- 自分たち用ツールとしては十分
- UDP でも扱いやすい
- 証明書管理などに比べて実装が軽い
- 許可外クライアントを弾ける

### 影響
- AuthRequest / AuthResponse が必要になる
- 設定ファイルに token 管理が必要になる
- secret の取り扱いに注意が必要になる

---

## ADR-013: ログは JSON Lines、メトリクスは switcher 表示とする

### 決定
ログは JSON Lines 形式の構造化ログとし、リアルタイム確認は switcher UI 上の簡易メトリクス表示で行う。

### 理由
- 後から解析しやすい
- 機械可読性が高い
- 配信中はその場で見えるメトリクスが必要
- Prometheus / Grafana などは MVP には重い

### 影響
- client / server / switcher ごとにログ出力を持つ
- run_id / client_id を全体で扱う
- UI 側に RTT / offset / fps / drop などの表示が必要になる

---

## ADR-014: app_version と protocol_version を分離する

### 決定
アプリのバージョンと通信仕様のバージョンを分離する。

### 理由
- UI変更や内部改善と、通信互換性の変更を分けたい
- protocol_version の不一致を明確に拒否したい
- app_version 差異は警告扱いにしたい

### 影響
- 認証メッセージに両方含める
- server は protocol_version を必ず検証する
- app_version はログや表示に使う

---

## ADR-015: リポジトリ構成は Cargo workspace の monorepo とする

### 決定
リポジトリは Cargo workspace の monorepo とし、apps と crates を分ける。

### 理由
- client / server / switcher の責務が明確
- 共通型や共通処理を crates に分離しやすい
- Rust の標準的な構成に寄せられる
- 将来の拡張やリファクタがしやすい

### 影響
- apps/client
- apps/server
- apps/switcher
- crates/protocol
- crates/config
- crates/logging
- crates/timebase
- crates/video-core
- crates/net-core
- crates/sync-core
- crates/ui-core

を基本構成とする

---

## ADR-016: Codex と GPT の共通認識は repo 内ドキュメントで管理する

### 決定
Codex と GPT の共通認識は、会話の自動共有ではなく、repo 内のドキュメントで管理する。

### 理由
- 永続的な外部記憶として機能する
- Codex は `AGENTS.md` を読める
- TODO や session-log を更新させれば引き継ぎしやすい
- 会話単体に依存しない

### 影響
- `AGENTS.md`
- `docs/requirements/project-overview.md`
- `docs/operations/todo.md`
- `docs/operations/session-log.md`

を最低限維持する
- Codex 作業後は TODO と session-log を更新する運用にする
